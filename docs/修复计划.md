  二、关键差异（架构 / 设计）
  
  G1 — _summarize_descriptions 名存实亡（中度风险）

  extraction/rag_extractor.py:64-74：
  async def _summarize_descriptions(entity_or_relation, descriptions, max_items=12):
      """当描述条目超过 max_items 时，用 LLM 将多条描述压缩为一条。"""
      # 改为保留原文，不做 LLM summarization
      return _merge_descriptions_raw(descriptions)
  注释说"超过 12 条时 LLM 压缩"，函数体直接 fallback 到原文拼接。RAGFlow general 模式同名函数实际会调 LLM。
  后果：高频实体（如"贵州茅台"）的描述会无限累加，properties.descriptions 字段越涨越大，影响向量检索性能。
  建议：要么删掉 max_items 参数明确"永不压缩"，要么补上真正的 LLM 调用并设置上限。
  
  G2 — 缺 gleaning 循环（高价值缺失）

  RAGFlow general 的核心 trick：
  extract → check 是否漏 → CONTINUE_PROMPT 再抽 → LOOP_PROMPT 决定停止
  最多 N 轮（默认 1 轮 + 1 次 LOOP 判断）
  我们 rag_extractor.py 里虽然 import 了 CONTINUE_PROMPT，但搜代码：grep CONTINUE_PROMPT extraction/：

  Searched for 1 pattern (ctrl+o to expand)

● gleaning 实现存在，但 RAGExtractor.__init__ 默认 max_gleanings=0（line 417），下游 extract_sync/extract_async 也是 max_gleanings=0（line 683/710）。等于完全关闭。实际 
  gleaning 不工作。RAGFlow ENTITY_EXTRACTION_MAX_GLEANINGS 默认是 1。

  G3 — 缺 LightExtractor（高 ROI）

  RAGFlow graphrag/light/graph_extractor.py 是为短文本设计的轻量版：
  - 单 prompt 不带 examples（example_number=2 内嵌）
  - 不做 gleaning
  - pack_user_ass_to_openai_messages 按 user/assistant 交替组织上下文
  
  公告（标题+一段话）和互动易 Q&A（一问一答）天然适合 light，硬塞 general 的 EXTRACTION_PROMPT（含 7 类禁止规则、3 类白名单、显式陈述原则）会让小模型困惑。我们
  ANNOUNCEMENT_EXTRACTION_PROMPT 看起来是 light 版的雏形，但仍走 RAGExtractor 同一套循环。

  G4 — 缺社区检测 / Mind Map（Phase 20 路线图欠款）

  RAGFlow graphrag/general/leiden.py + community_reports_extractor.py + mind_map_extractor.py 提供：
  - Leiden 算法发现实体社区
  - 给每个社区生成 LLM 摘要
  - mind_map 给单个实体生成层级摘要

  我们的 KGSearchEngine 已经有 _search_communities 但只是占位降级。

  G5 — 缺 figure_parser 但保留 figure 字段

  RAGFlow naive 模式会调 vision_figure_parser_pdf_wrapper(tbls=tables, sections=sections)，把图片让视觉 LLM 描述。我们丢弃了所有图片信息。投研中
  K线/产品照片确实信息少，这个差距可以接受。

  G6 — 缺 markitdown 通用文档转换

  DeerFlow 用 markitdown 把 PPT/Excel/Word 全转 Markdown 再走 PDF 同管道。RAGFlow 也支持 docx/xlsx/pptx/epub 各自的 parser。我们 extract_text_from_file 只支持 .pdf 和
  .txt。投研场景如果有人传 docx 路演稿/xlsx 财务表会直接拒绝。

  G7 — file_security.py 模块定义但 0 处调用

  grep -rn "file_security\|normalize_filename\|validate_path_traversal" backend/app
  # 全部命中都在 file_security.py 自己内部
  Phase 30 SUMMARY 写"新增 file_security.py（DeerFlow 路径安全模式）"，但调用方 file_storage.py / document_service.py / kg_extractor.extract_document(path) 都没接入。

  G8 — document_service.UPLOAD_DIR 仍然存在

  UPLOAD_DIR = Path(__file__).resolve().parent.parent.parent / "uploads"
  UPLOAD_DIR.mkdir(exist_ok=True)
  import 时立刻 mkdir。Phase 30 SUMMARY 说"移除上传功能"，但代码层还在创建上传目录。get_documents_by_ts_code / get_document_by_id / update_document_analysis
  三个查询函数还从 MongoDB uploaded_documents 读，这些是上传产物的查询接口，没有 producer 但仍然导出。

  G9 — ingestion/__init__.py 暴露错误的 API

  from app.knowledge.ingestion.pdf_parser import extract_pdf_sections
  from app.knowledge.ingestion.report_parser import extract_pages
  __all__ = ["extract_pdf_sections", "extract_pages"]
  但实际外部用的全是 extract_text_from_file（document_service）和 parse_report / parse_and_extract（report_parser）。extract_pdf_sections / extract_pages 是低级
  helper，不该作为公共 API。

  G10 — pdf_parser.py 与 report_parser.py 双套实现重复

  两份解析器：
  - pdf_parser.py:extract_text_from_pdf — 用 pdfplumber chars 分组、表格→markdown、PdfSection 类
  - report_parser.py:parse_report — 用 pdfplumber page.extract_text()、自己启发式分段、Chunk 类

  调用关系：
  - document_service.extract_text_from_file → pdf_parser.extract_text_from_pdf
  - kg_extractor → document_service.extract_text_from_file → 上面那条
  - report_parser.parse_report 与 parse_and_extract 看不到任何调用方

  grep -rn "parse_report\|parse_and_extract" --include="*.py"：

  Searched for 1 pattern (ctrl+o to expand)

● 证实：report_parser.py 整个文件没有外部调用方。除了 __init__.py 暴露 extract_pages 但也没人用。整 461 行死代码。

  三、Bug 清单（按严重度）
  
  🔴 阻塞 / 高数据正确性

  K1 — gleaning 关闭但 docstring 说"启用"
  rag_extractor.py:417 max_gleanings: int = 0；line 7 注释 "1. gleaning 循环（最多 N 轮追加抽取）"。文档与代码不符，实体召回率显著低于 RAGFlow general。
  修复：默认改为 1（RAGFlow 同款），让 docstring 与代码一致。

  K2 — _summarize_descriptions 假实现
  rag_extractor.py:64-74 注释说"超过 max_items 时 LLM 压缩"，函数体 fallback 拼接。
  后果：descriptions 字段无限增长。
  修复：要么删 max_items 参数（明确"始终保留原文"），要么补 LLM 调用。

  K3 — report_parser.py 461 行死代码
  0 处外部引用。ingestion/__init__.py 暴露的 extract_pages 也是孤儿。
  建议：直接删除 report_parser.py，简化 ingestion/__init__.py。

  K4 — document_service.UPLOAD_DIR 模块加载时 mkdir
  document_service.py:14-15：
  UPLOAD_DIR = Path(__file__).resolve().parent.parent.parent / "uploads"
  UPLOAD_DIR.mkdir(exist_ok=True)
  Phase 30 说"移除上传"，但模块 import 就建目录。
  修复：删 UPLOAD_DIR 和它的 mkdir。
  
  K5 — extract_text_from_file 接收 file_path: Path 但 kg_extractor 传 path
  kg_extractor.py:731：text = extract_text_from_file(path, ext) —— 用户输入路径未经 validate_path_traversal 校验，直接传入 pdfplumber。如果 path 包含 ../../../etc/passwd
  等，会直接读出文件内容。
  修复：在 extract_text_from_file 入口加 validate_file_path(path, allowed_base) 校验。

  K6 — file_security.py 整个模块 0 处调用
  存在但没被接入到 PDF 下载（file_storage.py）、PDF 解析（extract_text_from_file）、KG 抽取（extract_document）。
  修复：在三处入口接入。

  K7 — naive_merge 等价功能没有 overlap
  chunker.chunk_by_token 是硬切，前后 chunk 之间没有 overlapping。RAGFlow naive_merge 默认 overlapped_percent=0
  但参数化可调；研报中关键实体常在跨段处（"中芯国际新产能[换行]于 2026Q1 投产"），无 overlap 会导致关系丢失。
  修复：chunker 支持 overlapped_percent 参数。

  🟠 中（架构 / 一致性）

  K8 — ingestion/__init__.py 暴露低级 helper
  应该 export 的是 extract_text_from_file（公共入口），目前 export 的 extract_pdf_sections / extract_pages 没人用。

  K9 — 公告/互动易没有 light extractor
  两类轻文档与研报共用 EXTRACTION_PROMPT（含 7 类禁止规则），prompt token 浪费严重。

  K10 — pdf_parser._is_header_footer 用绝对像素位置判断
  top / page_height < 0.05 即认为页眉。研报常用大标题（如"年度策略报告"）正好落在前 5%，会被误判为页眉过滤掉。
  修复：结合"短文本 + 数字 + 重复出现"综合判断，仅靠位置不够。

  K11 — pdf_parser._table_to_markdown 不去重表头
  pdfplumber extract_tables 在跨页表格时每页都返回一遍表头，转 markdown 后会有多份表头行，污染 chunk。
  修复：跨页表格合并/表头去重。

  K12 — _clean_text 把所有单换行变空格
  text = re.sub(r"([^\n])\n([^\n])", r"\1 \2", text) 会把诗歌、代码、列表项中的换行也合并。研报里的列表项（"• 第一点[换行]• 第二点"）会被压成一行。
  修复：仅对"两边都是非标点中文/英文字符"的换行做合并。

  🟢 低（可读性 / 维护性）

  K13 — pdf_parser docstring 提到"PyPDF2 3.0" （report_parser.py:11）但代码已统一为 pdfplumber。
  K14 — extract_pdf_sections 的 max_pages: int = 1000 没用户控制入口，超大 PDF 默认全读。
  K15 — _detect_title 检测"居中"用 abs(x0 - center) < page_width * 0.15，但 PDF 中 x0 是文本块左边界，居中文本的 x0 应在 center - text_width/2 附近，不是
  center。该启发式实际不工作。
  K16 — extraction/chunker.py 与 report_parser.py 两套 Chunk 类型：NamedTuple Chunk(content, token_count, chunk_id, heading) vs dataclass Chunk(text, heading, 
  heading_level, chunk_type, page_start, page_end, token_count)。字段不一致让上下游调用混乱。

  ---
  结论：对标 RAGFlow，文件处理层（PDF / 文本提取 / 分块）的架构差异总体合理——投研研报基本是文本型 PDF，不需要 OCR/Vision/版面识别那一套重型流水线。但代码层有 6 个 P0 级
  bug，集中在：
  
  1. 核心 trick（gleaning）功能性关闭（K1）
  2. _summarize_descriptions 名存实亡（K2）
  3. 报告解析器整文件 461 行死代码（K3）
  4. 路径安全模块定义了但 0 处调用（K5 / K6）
  5. UPLOAD_DIR 仍在 import 时 mkdir（K4），与 Phase 30 SUMMARY 自相矛盾

  是否继续按 P0→P3 顺序修复？建议先做 K1（启用 gleaning）和 K3-K6（清理 + 接入 file_security），因为这几个直接影响数据正确性和安全。
