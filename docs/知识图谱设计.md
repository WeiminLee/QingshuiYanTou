# 09 - 投资知识图谱通用 Schema 设计
---

## 一、核心设计原则

### 四条铁律

| 铁律 | 含义 |
|------|------|
| **正交性** | 实体各司其职，关系不预定义类型 |
| **时序性** | 所有事实带有效时间窗，不覆盖历史，只追加 |
| **可溯源性** | 每条数据绑原始出处 source + weight |
| **克制扩展** | 最小 Schema，3 类实体 + 文本关系，业务逻辑外置 |

### 最重要的原则

```
图谱回答：谁 和 谁 有什么关系，置信度多少
推理层回答：这个关系意味着什么机会或风险
```

## 二、实体设计（3 类）

### 2.1 实体类型

```
Company  — 投资标的的核心锚点，所有关系的起点
Product  — 供应链/竞争的中间节点，可跨公司比较
Metric   — 量化指标，独立节点，必须有量化数值才能入库
```

> 2026-04-14 更新：删除 Event / Tech / Industry / Capacity 节点类型，
> 这些统一归入属性。Metric 作为独立节点（参考 Oxford 论文最佳实践）。

```json

  "valid_to": null

```

**Metric 入库强制规则**：
- `properties.value` 必须是数字
- `properties.unit` 必须是单位（%、亿元、万元、元、亿、万只、万吨等）
- 无数值的 Metric 不入库

## 三、关系设计（无类型标签）

### 3.1 核心设计

```
新设计：关系是一句自然语言陈述 + weight 置信度
```
**原因：**
1. LLM 判断关系类型时缺乏足够约束，容易误分类
2. 预定义类型无法覆盖投研文档的丰富语义
3. 自然语言陈述保留了原始信息的完整性

### 3.2 关系结构

```json
{
  "from_entity": "CO:300308_SZ",
  "to_entity": "CO:HUAWEI",
  "text": "中际旭创向华为交付800G光模块，月产能1.2万件",
  "weight": 1.0,
  "source_chunk": "第3章第2节",
  "source_file": "光通信产业链核心赛道全解析.md",
  "valid_from": "2024-01-01",
  "valid_to": null
}
```

### 3.3 weight 规则

| weight 值 | 含义 | 判断标准 |
|-----------|------|---------|
| 1.0 | 直接陈述 | chunk 中明确写明的陈述，LLM 仅转述，未增加新信息 |
| < 0.5 | LLM 推断 | LLM 根据多个线索推断出的关系，存在不确定性 |

**判断示例：**
- "中际旭创向华为交付800G光模块，月产能1.2万件" → weight = 1.0（原文原意）
- "中际旭创为华为核心供应商" → weight = 0.4（LLM 推断，非原文直接陈述）

### 3.4 同一实体对多关系规则

- **同一 chunk 内**：一对实体只产生一条关系陈述
- **不同 chunk 之间**：同一实体对可以有多个关系陈述（分别来自不同 chunk），各自独立
- **多维合并**：如果同一 chunk 中同时提到供货和股权两个维度，合并在一句话中

### 3.5 Neo4j 存储

```
所有关系统一为一种类型：RELATES

MATCH (a:Company)-[:RELATES {weight: 1.0}]->(b:Company)
RETURN a.name, b.name, r.text
```

Layer 3 查询时按 `weight >= 0.5` 过滤高置信关系。

## 五、Product 定期归一化

### 5.1 问题

LLM 每次抽取可能生成不同名称的同一产品：

```
"800G光模块"     → PROD:A3F2B1C9
"800G 光模块"    → PROD:B4E5F6A7
"800G光通信模块"  → PROD:C1D2E3F4
```

### 5.2 归一策略
**Step 1**：提取时保留原始名称

```json
{
  "entity_id": "PROD:A3F2B1C9",
  "name": "800G光模块",
  "properties": {
    "raw_names": ["800G光模块", "800G 光模块", "800G光通信模块"]
  }
}
```

## 六、时序建模
所有关系必须携带：

```
valid_from: date     # 关系生效起始日
valid_to:   date     # 关系失效日（null = 至今有效）
```

**严禁覆盖历史数据。更新时：**
1. 将旧边 `valid_to = update_date - 1`
2. 追加新边，`valid_from = update_date`
